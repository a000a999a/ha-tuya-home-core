Tuya HA Suite

Three HACS integrations for Home Assistant that share a single Tuya Cloud connection to control watering valves and monitor IP cameras — with real-time motion alerts and SD card health tracking.

Tuya Home Core v0.2.0 Tuya Watering v0.1.0 Tuya Cameras v0.2.0 HA ≥ 2024.6 HACS required

Overview

The suite is split into three focused integrations. Tuya Home Core authenticates once with the Tuya Cloud API and shares the connection, device list, and area map across all companions. The companion integrations declare it as a dependency and never need their own credentials.

🔑

Tuya Home Core

Shared Tuya Cloud credentials, device discovery, and area mapping. Required by both companions.

cloud_polling v0.2.0
💧

Tuya Watering

Local-first control of Tuya Zigbee water valves via tinytuya. Supports multiple valves per gateway, configurable DPS codes.

local_push v0.1.0
📷

Tuya Cameras

SD card monitoring, real-time MQTT motion alerts with email notifications per area, and manual SD format triggers.

cloud_push v0.2.0

Architecture

All three integrations run inside the Home Assistant process. There are no external servers, no helper scripts, and no systemd services required.

┌───────────────────────────────────────────────────────┐ │ Home Assistant Process │ │ │ │ ┌──────────────────────────────────────────────┐ │ │ │ tuya_home_core │ │ │ │ TuyaHomeAPI ←→ Tuya Cloud REST API │ │ │ │ TuyaHomeCoreCoordinator (refresh: N days) │ │ │ │ └─ device list + area map (shared) │ │ │ └──────────────┬───────────────────────────────┘ │ │ │ hass.data["tuya_home_core"] │ │ ┌───────┴────────┐ │ │ ┌──────▼──────┐ ┌──────▼───────────────────────┐ │ │ │tuya_watering│ │ tuya_cameras │ │ │ │ tinytuya │ │ CameraCoordinator │ │ │ │ local UDP │ │ └─ SD status poll │ │ │ │ switch.* │ │ TuyaMQTTBridge │ │ │ │ │ │ └─ AES-ECB motion events │ │ │ │ │ │ Notifier (SMTP email) │ │ │ └─────────────┘ └──────────────────────────────┘ │ │ │ └───────────────────────────────────────────────────────┘

Data flow

StepWhat happens
1. HA starttuya_home_core loads first (it's a declared dependency). Authenticates with Tuya Cloud, fetches device list and area map, stores them in hass.data["tuya_home_core"].
2. Companion loadtuya_watering and tuya_cameras read the shared API client and coordinator from hass.data — no extra credentials needed.
3. Scheduled refreshEach coordinator refreshes its data on a user-configurable interval (default 14 days). Manual refresh available via button entity.
4. Motion eventstuya_cameras MQTT bridge subscribes to Tuya's EU message queue. Motion DPS codes trigger email notifications to the configured area recipients.

Prerequisites

Tuya Developer Account

Create a Tuya IoT project

Go to iot.tuya.com, create a project, and link your SmartLife app account. Choose the Smart Home industry.

Note your credentials

From the project overview page, copy your Access ID (API Key) and Access Secret. These are entered once into Tuya Home Core.

Enable required APIs

In the project's API Products tab, ensure IoT Core, Device Status Notification, and Smart Home Basic Service are subscribed.

Link SmartLife devices

Your cameras and valves must be added to SmartLife homes. The integration reads home and area assignments directly from the Tuya Cloud API.

Email (Cameras only)

Motion and SD card alerts are sent via SMTP. Gmail works well — create an App Password (not your account password) under Google Account → Security → 2-Step Verification → App passwords.

Local network (Watering only)

Tuya Zigbee valves are controlled locally via tinytuya UDP. The HA host must be on the same network as the Zigbee gateway. You will need the gateway's local IP, local key, and the sub-device's CID — all obtainable from the Tuya IoT platform device list or via tinytuya's scan tool.

⚠️

Never call gw.status() on a Tuya Zigbee gateway device. This returns a device22 error, hangs the socket, and wastes API quota. Only call .status() on sub-devices.

Tuya Watering

Local-first valve control with zero cloud dependency during watering — commands go directly to the Zigbee gateway over your LAN via tinytuya.

One entry per valve group

Create one config entry for each Zigbee gateway. Each entry can manage multiple sub-device valves. The entry name becomes the group label (e.g. Terrasse, Keller).

Adding valves

After creating the entry, go to Configure → Add valve. Fill in the valve details:

FieldDescriptionDefault
nameDisplay name for the switch entity—
device_idTuya sub-device ID—
gateway_idTuya gateway device ID—
gateway_ipLocal IP of the Zigbee gateway—
gateway_keytinytuya local key (32 hex chars)—
sub_cidSub-device CID (Zigbee node identifier)—
default_durationWatering duration in seconds120
dps_durationDPS code for timer duration104
dps_triggerDPS code to start watering116
dps_stopDPS code for manual stop1
ℹ️

The DPS defaults (104 / 116 / 1) match the GIEX GX-02BT valve. Other Tuya valves may use different codes — check your device's DPS map on the Tuya IoT platform.

How watering works

Sequence (tinytuya local)
# 1. Set timer duration via DPS 104
sub.set_value(104, duration_seconds)
# 2. Wait 0.5s between DPS commands
# 3. Trigger watering via DPS 116
sub.set_value(116, True)
# Valve closes automatically after duration expires

What it creates

switch
{Valve name}
One switch per configured valve. Turning on starts watering for the configured duration.

Safety notes

🚫

Always set socketTimeout(8) before any tinytuya command. Without it, an unreachable gateway hangs the HA event loop indefinitely.

Do not call gw.status() on the gateway object. Call .status() on the sub-device only.

Tuya Cameras

SD card monitoring, real-time motion alerts via Tuya MQTT, and per-area email notification routing — all without AI or external services.

Setup wizard — three phases

Select Tuya account

If you have a single Tuya Home Core entry it is selected automatically.

SMTP / email settings

Enter your SMTP host, port, sender address, and App Password. Gmail: smtp.gmail.com:587, use an App Password, not your login password.

Per-area recipients

For each area that has cameras, enter semicolon-separated recipient addresses for human alerts (motion) and tech alerts (SD card, camera offline). Leave blank to disable alerts for that area.

Motion alert pipeline

Real-time motion events arrive via the Tuya MQTT message queue (no polling). The bridge decrypts each message and fires an email.

Event flow
Tuya Camera
  └─→ MQTT broker (ssl://m1.tuyaeu.com:8883)
        └─→ TuyaMQTTBridge  (HA async task)
              ├─ decrypt: AES-128-ECB, key = mqtt_password[8:24]
              ├─ parse: DPS code movement_detect_pic / initiative_message
              ├─ lookup: device → area → recipients
              └─→ Notifier.send_alert()  (SMTP)

SD card monitoring

On each scheduled refresh (default every 14 days, or on manual button press), the coordinator polls each camera's SD card status via the Tuya Cloud API and creates sensor entities. An alert email is sent when usage exceeds the configured threshold (default 90%).

Technical Reference

MQTT bridge — connection and decryption

The TuyaMQTTBridge uses paho-mqtt over TLS to subscribe to Tuya's EU message queue. Credentials are fetched once at startup and refreshed automatically before expiry.

MQTT credentials endpoint

HTTP
POST /v1.0/open-hub/access/config
Body: {
  "uid":       "<YOUR_TUYA_UID>",
  "link_id":   "<YOUR_ACCESS_ID>",
  "link_type": "mqtt",
  "topics":    "device"
}

Broker

Config
url:      ssl://m1.tuyaeu.com:8883
topic:    cloud/token/in/{session_id}
expiry:   7200 seconds  →  reconnect at T-10 min

Message decryption

Python
from cryptography.hazmat.primitives.ciphers import Cipher, algorithms, modes
# Key = bytes 8–23 of the MQTT session password (changes every session)
key  = mqtt_password[8:24].encode()
# Cipher: AES-128-ECB with PKCS7 padding
cipher = Cipher(algorithms.AES(key), modes.ECB())

Coordinator pattern — shared device data

All three integrations use HA's DataUpdateCoordinator. The core coordinator fetches the device list; companions read from it without extra API calls.

Python
# Cameras coordinator reads core data — no extra API call for device list
async def _async_update_data(self):
    core_data = self._core_coordinator.data or {}
    devices   = core_data.get("devices", [])
    area_map  = core_data.get("areas", {})
    cameras   = self._camera_api.cameras_from_devices(devices, area_map)
    # Only the SD status poll hits the cloud (one call per camera)
    for cam in cameras:
        sd = self._camera_api.get_sd_status(cam["id"])
        ...

API Quota

Tuya's Trial Edition (free tier) provides a limited monthly quota of API calls. This suite is specifically designed to stay well within it.

OperationCalls/month (default)Notes
Device list refresh~2–4Scheduled every 14 days + occasional manual presses
SD status poll~2 × N camerasOnce per refresh cycle, one call per camera
MQTT credentials~12–20One call per session; refreshed every ~2 hours
SD format (manual)On demandOnly when explicitly triggered by user
⚠️

Tuya Trial Edition quotas reset on the first day of each calendar month. If the quota is exhausted (HTTP 29002002 error), the coordinator falls back to its last cached data and logs a warning. No data is lost.

To stay within Trial limits: keep the auto-refresh interval at 7 days or more, and use the manual Refresh button only when actually needed.

Extending

AI / YOLO motion detection

The MQTT bridge calls Notifier.send_alert() on every motion event. To add AI filtering (e.g. human detection with YOLOv8), intercept between the MQTT message and the notifier call:

Python sketch
# In mqtt_bridge.py — extend _handle_motion_event():
image_bytes = await self._fetch_motion_image(event)
if image_bytes:
    label, conf = yolo_model.predict(image_bytes)
    if label != "person" or conf < 0.6:
        return  # suppress non-human alerts
await self._notifier.send_alert(...)

YOLOv8 nano (ultralytics package) runs comfortably on a CPU-only HA host for single-frame inference at ~100–300ms per image.

Additional sensor types

The CameraCoordinator data dict already contains online (bool), sd_status (str), and all SD metrics. Adding a new sensor is a matter of subclassing SensorEntity in sensor.py and pointing it at the existing coordinator data key.

Notification channels

The Notifier class in notify.py is the single output point. Swap the SMTP backend for Telegram, Pushover, or any other channel by implementing the same send_alert(subject, body, area) interface.

Camera dashboard

The SD sensor entities are standard HA sensors — they work out of the box with any Lovelace card. A grid of gauge cards showing SD usage per camera, grouped by area, can be built with no custom code.
